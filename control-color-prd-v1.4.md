**第1章 项目概述**

**1.1 项目名称**

Control-Color —— 基于浏览器的图片色彩处理工具

**1.2 项目背景**

现有的图片色彩处理工具要么门槛较高（需要专业软件），要么功能过于单一。Control-Color 专注于色彩处理，提供配色提取、分析、迁移和创建配色方案的能力，同时提供用户系统支持收藏与个性化配置。

**1.3 目标用户**

UI/UX 设计师

摄影师 / 后期制作人员

前端开发者

色彩爱好者 / 学习者

**1.4 核心价值主张**

专注色彩：深入色彩处理领域，提供专业级工具

智能降低门槛：通过智能推荐和预设方案，让非专业用户也能轻松创建美观配色

即时反馈：所见即所得，实时预览处理效果

易用性：基于浏览器，无需安装，开箱即用

个性化：用户系统支持收藏、偏好配置和数据同步


**1.5 核心差异化定位**

以下是 Control-Color 与现有竞品的核心差异化定位：

**配色迁移保护素描关系：** 在将配色方案应用到目标图片时，通过亮度保护算法确保原图的明暗层次（素描关系）不被破坏。这是与现有竞品最核心的区别——大多数调色工具在改变色彩时会影响亮度分布，导致画面失去原有的光影结构。Control-Color 在 lαβ 色彩空间中仅调整 α（黄蓝）和 β（红绿）色度通道，保持 l（亮度）通道不变，从而在改变色调的同时完整保留原图的素描关系。

**直观对比与实时调整：** 左右并排对比视图，所有调整实时预览，用户可以精确控制迁移强度。

**色彩理论驱动的智能推荐：** 基于 OKLCH 色彩空间的纯规则算法，为没有色彩理论知识的用户提供专业级配色建议。

**纯前端处理：** 无需上传图片到服务器，隐私安全，无网络延迟。

**一站式色彩工具：** 提取、迁移、编辑、推荐、导出全流程在一个工具中完成。

**第2章 功能需求**

**2.1 图片配色提取 (Color Palette Extraction)**

功能描述：从上传的图片中自动提取主要配色方案。

**2.1.1 提取算法**

K-Means 聚类（默认算法）

Median Cut MMCQ（可选算法）

提取数量：3 \~ 12 个（默认 6 个）

按像素占比排序

**2.1.2 交互设计**

拖拽上传或点击选择（支持 PNG / JPG / WebP）

色块卡片展示（显示 HEX / RGB / HSL 数值）

点击色块复制色值

支持删除颜色

补充颜色：通过智能推荐算法，基于已有配色方案分析缺失的色彩区域（如缺少互补色、缺少高饱和度色等），推荐 3\~5 个合适的新颜色供用户选择添加

**2.1.3 图片来源**

拖拽文件上传

点击选择文件

粘贴剪贴板图片（Ctrl+V，支持 bitmap 和 HTML 中的图片）

输入图片 URL

**2.1.4 图片上传体验**

上传进度：大文件上传到 Supabase Storage 时显示进度条（百分比 + 已上传/总大小）

URL 图片加载：通过后端代理转发（避免 CORS 问题），加载失败时提示"无法加载该图片，请检查 URL 或尝试本地上传"

粘贴兼容性：支持从截图工具、文件管理器、网页图片直接 Ctrl+V 粘贴，自动识别剪贴板中的图片数据

加载失败降级：图片加载失败时显示错误占位图 + "重新上传"按钮，不阻塞其他功能使用

**2.2 配色迁移 (Color Transfer)**

功能描述：将 paletteStore 中的当前配色方案应用到目标图片上，同时保护目标图的素描关系（明暗层次）不被破坏。迁移的输入是 paletteStore.colors（可来自提取、预设、手动编辑等任何来源），而非直接读取源图片素。

**2.2.1 迁移原理**

核心流程：提取源图配色 → 计算源图与目标图在 lαβ 色彩空间中的统计量差异 → 仅调整目标图的 α（黄蓝）和 β（红绿）色度通道 → 保持目标图 l（亮度）通道不变 → 确保目标图素描关系完整保留。

**2.2.2 迁移算法**

快速模式：Reinhard 算法（在 lαβ 色彩空间中，仅对 α 和 β 色度通道进行均值和标准差匹配，保持 l 亮度通道不变，确保素描关系完整保留）

精确模式：直方图匹配（在 lαβ 空间中对 α 和 β 通道分别进行直方图匹配，l 通道保持不变）

**2.2.3 迁移强度控制**

滑块范围：0% \~ 100%

实现方式：原图与迁移结果的 alpha 混合。0% 为原图，100% 为完全迁移结果，中间值按比例线性混合。

计算公式：`output = original × (1 - intensity) + transferred × intensity`

**2.2.4 交互流程**

准备配色方案（提取/预设/手动编辑）→ paletteStore 自动更新 → 上传目标图（右框）→ 实时预览迁移效果 → 调整强度滑块 → 导出

**2.2.5 边界情况处理**

- 源图与目标图尺寸差异大时：迁移算法基于像素颜色统计量，不受尺寸影响，正常处理
- 目标图为纯黑白图时：α/β 通道本身接近零，迁移效果有限，UI 提示"目标图色彩信息较少，迁移效果可能不明显"
- 源图与目标图为同一图片时：迁移结果与原图一致（统计量完全匹配），正常显示

**2.3 预设配色方案 (Preset Palettes)**

功能描述：基于色彩理论的预设配色方案，算法生成，不涉及第三方版权。

**2.3.1 配色规则**

|          |             |               |
| -------- | ----------- | ------------- |
| **配色类型** | **角度关系**    | **说明**        |
| 单色系      | 同色相         | 相同色相，不同亮度/饱和度 |
| 互补色      | 180°        | 色相环上对立的两种颜色   |
| 类似色      | ±30°        | 相邻色相的颜色组合     |
| 三色组      | 120°        | 色相环上等距三点      |
| 分裂互补色    | 150° + 210° | 互补色的变体形式      |
| 四色组      | 90°         | 色相环上等距四点      |

**2.3.2 实现方式**

使用 OKLCH 色彩空间计算

基于基础色生成各类型配色

每种配色规则生成固定数量的颜色（默认 5 色，可在偏好设置中配置 3~8 色）

随机配色按钮

一键应用到目标图片

使用 OKLCH 色彩空间计算

基于基础色生成各类型配色

随机配色按钮

一键应用到目标图片

**2.4 用户自主配色 (Custom Color Editing)**

功能描述：丰富的配色编辑工具。

色轮选择器：可视化色相选择，支持拖拽

多格式输入：HSL、OKLCH、HEX、RGB

滑块微调：色相、饱和度、亮度独立滑块

配色列表管理：添加、删除、拖拽重排序

实时预览

**2.5 智能配色推荐 (Smart Color Suggestions)**

功能描述：纯规则算法，根据用户当前已选择的颜色实时推荐和谐配色，降低色彩设计门槛。

**2.5.1 推荐算法规则**

输入：用户当前配色方案中的所有颜色

色彩理论推荐：基于用户选中的颜色，在 OKLCH 空间中计算互补色(±180°)、类似色(±30°)、三色组(±120°)、分裂互补色(±150°/±210°)

接近色推荐：基于用户选中颜色，生成色相相同但亮度/饱和度微调的变体（±10% 亮度、±15% 饱和度）

排序策略：与现有配色方案的感知距离排序，优先推荐差异适中的颜色

去重过滤：与已有配色方案中颜色过于接近的（OKLCH ΔE \< 5）自动过滤

输出：最多 8 个推荐颜色，附带推荐类型标签（“互补色”、“类似色”、“亮度变体”等）

**2.5.2 交互设计**

推荐区域在配色编辑区旁实时更新

一键采纳

推荐原因标注

可关闭

**2.6 灰度图转换 (Grayscale Conversion)**

平均值法

加权平均法（BT.601）

去饱和法

灰度图作为后续灰阶区域配色功能的基础。

**2.7 RGB 通道分离 (Channel Separation)**

通道分离显示

通道强度调节（0% \~ 200%）

通道混合（自定义权重，每个通道权重范围 -100% \~ 200%，三通道权重之和无需等于 1，输出值自动 clamp 到 0\~255）

通道交换

**2.8 灰阶区域配色 (Tonal Range Color Mapping)**

灰度直方图（0 \~ 255）

预设三区域：阴影 / 中间调 / 高光

自定义区域范围（拖拽边界）

应用指定颜色或渐变色

实时预览

**2.8.1 渐变色映射方式**

渐变色通过自定义曲线映射到灰阶区域：

- 每个区域可设置起始色和结束色，形成渐变
- 区域内部通过线性插值将灰度值映射到渐变色
- 区域边界处使用 20px 宽度的平滑过渡（sigmoid 混合），避免硬边
- 支持贝塞尔曲线编辑器调整映射曲线形状（类似 Photoshop 曲线工具）
- 曲线编辑器横轴为灰度值（0\~255），纵轴为输出色彩参数

**2.9 配色方案导出 (Export)**

**2.9.1 配色方案格式**

CSS 变量：生成 `:root { --color-1: #FF6B35; --color-2: ... }` 格式，变量名按序号命名，附带注释标注色值

Tailwind CSS 配置：生成 `theme.extend.colors` 对象，可直接合并到 `tailwind.config.js`

JSON：包含完整配色数据（name、colors 数组、sourceType、tags），与 palettes 表 colors 字段结构一致

SVG 色块：生成水平排列的色块 SVG 图片，每个色块下方标注 HEX 值，宽高可配置

**2.9.2 图片导出**

处理后图片（PNG / WebP）：导出应用配色迁移/灰阶区域配色/通道调整后的图片，分辨率为原图分辨率

**2.9.3 批量导出**

一键导出所有收藏：将用户全部收藏的配色方案打包导出为 JSON 文件，包含所有收藏的完整数据

**2.10 图片预处理 (Image Preprocessing)**

在配色提取或迁移之前，对图片进行基础预处理操作。

**2.10.1 支持的操作**

裁剪：自由裁剪或按比例裁剪（1:1、4:3、16:9 等）

旋转：90° 顺时针/逆时针旋转，自由角度旋转

翻转：水平翻转、垂直翻转

**2.10.2 交互设计**

预处理操作在图片上传后、色彩处理前进行

提供简单的裁剪框和旋转控件

预处理结果替换原图进入后续流程

预处理不可撤销（重新上传恢复原图）

**2.11 功能模块依赖关系**

各功能模块之间的依赖和流转关系：

```
图片上传（入口）
  ├── 配色提取 → paletteStore（独立功能，无前置条件）
  ├── 配色迁移 ← paletteStore + 目标图（需要先有配色方案）
  ├── 预设配色 → paletteStore（独立功能）
  ├── 用户自主配色 → paletteStore（独立功能）
  ├── 智能推荐 → paletteStore（需要已有配色方案作为输入）
  ├── 灰度转换 → 灰阶区域配色的前置条件
  │     └── 灰阶区域配色 ← paletteStore（需要先转换为灰度图）
  ├── RGB 通道分离（独立功能）
  └── 图片预处理（上传后、色彩处理前的可选步骤）
```

独立功能（无前置条件）：配色提取、预设配色、用户自主配色、RGB 通道分离

依赖 paletteStore 的功能：配色迁移、灰阶区域配色、智能推荐

有前后依赖的功能：灰度转换 → 灰阶区域配色

**3.2 用户认证**

**3.2.1 登录方式**

邮箱密码：注册 + 登录，支持密码重置

第三方社交登录：Google、GitHub 一键授权

**3.2.2 安全策略**

密码策略：最少 8 个字符，建议包含大小写字母和数字（不强求特殊字符）

登录安全：
- 连续 5 次登录失败后锁定 15 分钟
- 登录成功后发送邮件通知（可关闭）

Session 管理：
- Access Token 有效期 1 小时，Refresh Token 有效期 7 天
- Refresh Token 自动轮换（每次使用后生成新的）
- 用户登出时清除所有 Token

API 频率限制：
- 登录/注册接口：同一 IP 每分钟最多 10 次请求
- 数据接口：同一用户每分钟最多 60 次请求
- Storage 上传：同一用户每分钟最多 5 次请求

**6.6 首页流星配色特效 (Homepage Meteor Effect)**

首页背景的交互式视觉特效，用户点击空白背景区域时，随机生成配色方案并以流星坠落效果展示，用户可捕获流星收藏配色。

**6.6.1 交互流程**

1. 用户点击首页空白背景区域（非 UI 元素区域）
2. 系统随机生成一个配色方案（4~6 色）
3. 从窗口左上区域随机位置生成一颗流星，使用配色方案中的颜色
4. 流星带着色彩尾迹向右下方坠落（1.5~2.5s）
5. 坠落过程中伴有轻微合成音效（Web Audio API 生成，无需加载音频文件）
6. 用户可在流星坠落过程中点击捕获流星
7. 捕获后流星停止坠落，弹出配色方案面板（色块 + "收藏此配色" 按钮）
8. 未被捕获的流星自然消散，配色色块在底部短暂浮现后淡出
9. 每次点击有 500ms 冷却间隔，同时最多存在 5 颗流星

**6.6.2 流星视觉效果**

| 要素 | 说明 |
| ---- | ---- |
| 起始位置 | 窗口左上区域随机位置（x: 5%~30%, y: 0%~10%） |
| 运动轨迹 | 向右下方斜落，随机偏移角度 -15°~+15° |
| 尾迹效果 | 渐变拖尾，颜色从配色方案中循环取色，从亮到透明 |
| 粒子散落 | 尾部散落微小粒子，增强视觉层次 |
| 生命周期 | 1.5~2.5s 后自然消散 |
| 捕获反馈 | 点击后流星放大、定格，伴随轻微缩放动画 |
| 文字碰撞 | 流星经过时撞击排开页面上的可见文字，文字向两侧散落后弹簧回弹归位 |

**6.6.2.1 文字碰撞排开效果**

流星坠落过程中经过页面文字区域时，文字字符被流星"撞开"，向两侧散落，流星通过后通过弹簧物理动画自然回归原位。

视觉效果描述：
- 流星接近文字时，字符开始受到排斥力影响，微微偏移
- 流星经过字符时，字符被向两侧弹开，距离与流星距离成反比
- 流星远离后，字符通过弹簧动画（stiffness + damping）自然回弹到原位
- 回弹过程中有轻微过冲（overshoot），增强物理感
- 碰撞过程中字符保持可读性，不旋转、不缩放、不透明度变化

参与碰撞的文字范围：首页所有可见文字（标题、副标题、功能介绍、按钮文字等）。

不参与碰撞的元素：流星特效层本身的粒子、图标、Logo 图形部分。

**6.6.3 配色生成策略**

每次点击从以下策略中随机选择一种：

| 策略 | 说明 | 触发概率 |
| ---- | ---- | -------- |
| 随机色相 + 色彩理论 | 随机选一个色相，生成互补/类似/三色组 | 40% |
| 完全随机 | OKLCH 空间随机采样，饱和度 50%~80%，亮度 50%~75% | 30% |
| 趋势配色 | 从预设趋势配色库中随机选取 | 20% |
| 渐变序列 | 基于一个起始色生成渐变色序列 | 10% |

**6.6.4 捕获与收藏**

**6.6.5.1 文字测量与渲染（@chenglou/pretext）**

使用 @chenglou/pretext 库实现首页文字的精确测量和 Canvas 渲染，作为流星碰撞效果的基础。

文字处理流程：
1. 页面加载时，使用 `prepareWithSegments(text, font)` 对首页所有可见文本进行预处理
2. 使用 `layoutWithLines(prepared, maxWidth, lineHeight)` 获取每行文本的精确布局信息
3. 从布局结果中提取每个字符的位置（x, y）、宽度（width）和行信息
4. 将文字渲染到 Canvas 上（而非 DOM 文本），使文字与流星在同一渲染层
5. 每帧更新时，根据流星位置计算字符的排斥力偏移量
6. 使用弹簧物理模型（stiffness: 300, damping: 20）计算字符回弹动画

pretext 使用范围：仅首页流星特效的文字测量，其他页面仍使用 DOM 文本渲染。

性能优化：
- 仅对视口内可见的字符进行碰撞计算
- 使用空间哈希（Spatial Hash）加速流星与字符的碰撞检测
- 字符偏移量使用 TypedArray 存储，减少 GC 压力
- 文本预处理结果缓存，窗口 resize 时重新计算

**移动端差异化适配**

移动端屏幕较小、文字较少，对流星碰撞效果做以下调整：
- 碰撞影响半径缩小为桌面端的 70%（减少在小屏幕上的视觉干扰）
- 弹簧参数降低刚度：stiffness 200, damping 25（移动端动画更柔和）
- 移动端默认关闭文字碰撞效果（可在设置中手动开启），以保证核心功能性能
- 移动端流星最大数量限制为 3 颗（桌面端为 5 颗）

**7.2 Web Worker 拆分策略**

放入 Worker 的计算任务：

直方图匹配

直方图计算

灰度转换

主线程保留：UI 交互、Canvas 渲染、状态管理

**7.2.1 Worker 通信协议**

数据传输格式：使用 Transferable Objects（ArrayBuffer）传递图片像素数据，避免结构化克隆的性能开销。

通信协议：
- 主线程 → Worker：`{ type: 'task_name', payload: { imageData: ArrayBuffer, width: number, height: number, params: object } }`
- Worker → 主线程：`{ type: 'task_name_result', payload: { resultData: ArrayBuffer, width: number, height: number } | { type: 'error', message: string }`

大图传输优化：图片数据通过 Transferable Objects 零拷贝传输，传输完成后主线程的 ArrayBuffer 自动 detached，不会阻塞主线程。

Worker 池：创建 2\~4 个 Worker 实例的池（navigator.hardwareConcurrency 自适应），任务队列自动分配。

**7.2.2 并发任务管理**

任务队列策略：
- 同类型任务排队执行（如连续调整迁移强度，取消前一个未完成的任务，只执行最新的）
- 不同类型任务可并行（如提取配色和灰度转换可同时进行）
- 任务优先级：用户主动触发的操作 > 自动预览计算

取消机制：
- 用户切换功能模块时，自动取消当前模块未完成的 Worker 任务（通过 `worker.terminate()` 或发送取消消息）
- 滑块拖动过程中使用 debounce（100ms），避免频繁触发计算
- 组件卸载时清理关联的 Worker 任务，防止内存泄漏

**7.8 关于 @chenglou/pretext**

@chenglou/pretext 用于首页流星特效中的文字精确测量和 Canvas 渲染，是流星碰撞排开文字效果的技术基础。

使用范围：仅首页流星特效（其他页面使用 DOM 文本渲染）。

使用方式：
- `prepareWithSegments(text, font)` 预处理文本，获取字符分段信息
- `layoutWithLines(prepared, maxWidth, lineHeight)` 获取每行布局和字符位置
- 从布局结果提取字符坐标，渲染到 Canvas 并参与碰撞计算

已有 React 集成包 `react-pretext-editor` 可用。建议在 P2 阶段集成。

**7.9 浏览器标签页可见性处理**

使用 `document.visibilityState` API 监听标签页可见性变化：

- 切换到后台时：
  - `requestAnimationFrame` 自动暂停（浏览器行为），流星动画自然停止
  - 正在进行的 Web Worker 计算继续执行（不取消，避免用户切回时需重新等待）
  - 流星生命周期计时暂停（记录离开时间，恢复时扣除暂停时长）

- 恢复可见性时：
  - 重新启动动画循环
  - 检查 Worker 计算是否已完成，若完成则更新 UI
  - 流星从暂停位置继续生命周期

**配色方案状态流转规则**

- 配色提取完成 → 自动替换 paletteStore 中的 colors，sourceType 设为 `extraction`
- 预设配色应用 → 替换 paletteStore，sourceType 设为 `preset`
- 用户手动编辑颜色 → 更新 paletteStore 中对应颜色，sourceType 变为 `custom`
- 智能推荐采纳 → 添加到 paletteStore.colors，sourceType 变为 `custom`
- 流星配色收藏 → 创建新的收藏记录（不修改当前 paletteStore）
- 迁移操作 → 读取 paletteStore.colors 作为迁移输入，不修改 paletteStore

- 删除操作使用软删除（deleted_at 标记），同步时优先处理删除操作

**第11章 竞品分析**

**11.1 竞品概览**

| 竞品 | 定位 | 优势 | 不足 |
| ---- | ---- | ---- | ---- |
| Adobe Color | 专业配色工具 | 生态完善、与 Adobe 全家桶集成 | 需付费、功能偏配色生成、无图片配色迁移 |
| Coolors | 快速配色生成 | 操作简单、社区方案丰富 | 仅配色生成、无图片处理能力 |
| Color Hunt | 配色方案社区 | 方案质量高、灵感来源 | 仅浏览收藏、无图片处理 |
| Palette.fm | AI 配色迁移 | AI 驱动、效果惊艳 | 黑盒算法不可控、无法精细调整、可能破坏素描关系 |

**11.2 核心差异化**

1. **素描关系保护**（最核心差异）：配色迁移时保持原图明暗层次不变，通过 lαβ 色彩空间的亮度通道隔离实现。Palette.fm 等竞品无法做到这一点。
2. **直观对比与精细控制**：左右并排实时对比 + 迁移强度滑块，用户可精确控制效果，而非"一键生成不可调"。
3. **色彩理论驱动**：基于 OKLCH 的纯规则推荐，透明可解释，非黑盒 AI。
4. **纯前端处理**：无需上传图片到服务器，隐私安全，无网络延迟。
5. **一站式色彩工具**：提取、迁移、编辑、推荐、导出全流程在一个工具中完成。

**第18章 术语表**

| 术语 | 英文 | 定义 |
| ---- | ---- | ---- |
| OKLCH | OK Lab Chroma/Hue | 一种感知均匀的色彩空间，O 表示光感知对齐，K 表示黑点，LCH 表示亮度/色度/色相。适合用于色彩计算和推荐 |
| lαβ 色彩空间 | Luminance alpha-beta | Reinhard 色彩迁移算法使用的色彩空间。l 为亮度通道，α 为黄蓝色度通道，β 为红绿色度通道。三个通道近似去相关 |
| K-Means 聚类 | K-Means Clustering | 一种无监督学习算法，将数据点分为 k 个簇。在本项目中用于将图像像素按颜色聚类，提取主要配色 |
| Median Cut | Median Cut (MMCQ) | 中值切割算法，通过递归将颜色空间二分来量化颜色，常用于调色板提取 |
| ΔE (Delta E) | Delta E | 色差单位，表示两种颜色在人眼中的感知差异。ΔE < 1 几乎不可分辨，ΔE < 5 为可接受差异 |
| Reinhard 算法 | Reinhard Color Transfer | 由 Erik Reinhard 等人提出的色彩迁移算法，通过匹配源图和目标图在 lαβ 空间中的统计量（均值和标准差）实现色彩风格迁移 |
| 直方图匹配 | Histogram Matching | 一种图像处理技术，通过调整图像的灰度/颜色分布使其与参考图像的分布一致 |
| BT.601 | ITU-R BT.601 | 国际电信联盟定义的标准，规定了将 RGB 转换为灰度的加权系数（0.299R + 0.587G + 0.114B） |
| 素描关系 | Tonal Relationship | 图像中明暗层次的相对关系，即光影结构。保持素描关系意味着在改变色彩时不改变像素的相对亮度 |
| WCAG | Web Content Accessibility Guidelines | W3C 发布的网页内容可访问性指南，定义了不同级别的无障碍标准 |
| RLS | Row Level Security | Supabase/PostgreSQL 的行级安全策略，确保用户只能访问自己的数据 |
| PWA | Progressive Web App | 渐进式 Web 应用，支持离线使用、安装到桌面等原生应用特性 |
| Service Worker | Service Worker | 浏览器后台脚本，用于拦截网络请求、管理缓存，实现 PWA 离线功能 |
| BaaS | Backend as a Service | 后端即服务，提供现成的后端功能（认证、数据库、存储），无需自建后端 |
| Design Token | Design Token | 设计令牌，将设计决策（颜色、间距、字号等）抽象为可编程的变量 |

**第19章 验收标准**

**2.1 配色提取**

- AC-1：上传 4000x3000 PNG 图片后，2s 内显示 6 个提取颜色（默认设置）
- AC-2：提取结果按像素占比从高到低排列，每个颜色显示 HEX/RGB/HSL 数值
- AC-3：点击色块可复制 HEX 值到剪贴板，并显示 Toast 提示
- AC-4：支持拖拽文件、Ctrl+V 粘贴、URL 输入三种图片来源
- AC-5：可手动删除提取颜色或添加新颜色

**2.2 配色迁移**

- AC-1：迁移后的图片与源图的亮度分布一致（SSIM 亮度分量 > 0.95），素描关系未被破坏
- AC-2：迁移强度滑块从 0%（原图）到 100%（完全迁移）连续可调
- AC-3：左右并排对比视图同步缩放和滚动
- AC-4：快速模式实时预览无卡顿（> 30fps），精确模式在 3s 内完成
- AC-5：导出的处理图片为原图分辨率，非缩放后的预览尺寸

**2.3 预设配色方案**

- AC-1：选择基础色后，在 500ms 内生成 6 种配色规则对应的方案
- AC-2：所有配色在 OKLCH 空间中计算，色相角度误差 < 1°

**20.1 公开 API 接口**

提供 RESTful API，允许第三方开发者集成配色提取和迁移功能。

**20.1.1 API 端点规划**

| 端点 | 方法 | 说明 | 认证 |
| ---- | ---- | ---- | ---- |
| `/api/v1/extract` | POST | 上传图片，返回提取的配色方案 | API Key |
| `/api/v1/transfer` | POST | 上传源图和目标图，返回迁移结果 | API Key |
| `/api/v1/palettes` | GET | 获取公开配色方案列表 | 无需认证 |
| `/api/v1/palettes/:id` | GET | 获取单个公开配色方案详情 | 无需认证 |

**20.1.2 API 设计规范**

- 请求/响应格式：JSON
- 图片传输：Base64 编码（单张图片限制 20MB，与主产品一致）或提供公开 URL
- 认证方式：API Key（通过 `Authorization: Bearer <key>` 头传递）
- 频率限制：免费层每分钟 30 次请求，付费层每分钟 300 次
- 版本管理：URL 路径中包含版本号（`/api/v1/`），后续版本通过 `/api/v2/` 向后兼容
- 错误响应：统一格式 `{ error: { code: string, message: string } }`

**20.1.5 API 与前端产品的关系说明**

API 接口与前端产品的定位不同，二者并行存在：

- **前端产品**：纯浏览器端处理，图片不上传到服务器，面向终端用户（1.5 核心差异化定位）
- **API 接口**：服务端处理，需要上传图片，面向第三方开发者集成使用

API 的图片上传仅用于服务端计算，处理完成后立即丢弃，不存储用户原图，与前端产品的隐私保护原则一致。

**20.1.3 API Key 管理**

- 用户在个人中心生成和管理 API Key
- 每个 Key 可设置名称和权限范围（仅读取 / 读取+写入）
- 支持撤销和重新生成
- 免费用户最多创建 2 个 Key，未来可通过付费升级

**20.1.4 开源协议**

项目采用 MIT 协议开源，允许自由使用、修改和分发。
